# 문서 피드백 및 성장 제안

---

## 1. 자기소개서 피드백

### 1-1. 강점

1. **커리어 스토리가 자연스럽게 이어짐**  
   - 학부 → 첫 회사(웹 FE) → 두 번째 회사(모바일 RN) → 프리랜서/사이드 → 목표까지 흐름이 명확합니다.  
   - “사용자에게 보이는 제품을 만들고 싶다 → 웹/앱 선택”이라는 동기가 설득력 있습니다.

2. **구체적인 성과와 수치 제시**  
   - LCP 3.2s → 1.8s, 이탈률 15% 감소, 코드 중복 30% 감소, 신규 기능 개발 속도 20% 향상 등은 매우 좋은 포인트입니다.  
   - 스타트업/서비스 회사에서 바로 이해할 수 있는 언어입니다.

3. **협업·커뮤니케이션에 대한 인식이 성숙함**  
   - 매주 데모 세션 제안, 기획/디자인과 사용자 플로우 중심 논의 등 “제품 팀의 동료”로서의 태도가 잘 드러납니다.  
   - 이후 포트폴리오의 “협업 & 업무 방식”과도 일관성이 있어 신뢰감을 줍니다.

4. **자기 성찰과 개선 의지가 드러남**  
   - 꼼꼼함이 장점이지만, 우선순위 조정이 필요하다는 자기 인식이 좋습니다.  
   - “단순 기능 구현자”가 아니라 “문제를 이해하고 해결하는 개발자”가 되고 싶다는 목표가 명확합니다.

5. **기술 스택과 실제 사용 맥락이 연결되어 있음**  
   - React, TS, React Query, Next.js, RN, Node.js 등 실제로 어디에 어떻게 썼는지가 자기소개서와 포트폴리오 전체에서 일관되게 나옵니다.

---

### 1-2. 개선 포인트

1. **지원 회사/직무에 대한 ‘맞춤형’ 메시지가 부족**  
   - 현재는 “귀사에서 웹/앱 개발자로 기회를 주신다면” 정도로만 언급되어, 어떤 회사든 복붙 가능한 느낌을 줍니다.  
   - 지원 회사의 서비스/도메인과 연결된 문장이 추가되면 설득력이 크게 올라갑니다.

2. **핵심 메시지가 다소 분산됨**  
   - 경험/가치관/기술/목표가 골고루 들어가 있지만, 문단마다 “이 문단의 핵심이 무엇인지”가 조금 흐립니다.  
   - “나는 어떤 유형의 개발자인가?”를 1~2문장으로 압축한 후, 나머지는 그걸 뒷받침하는 구조로 재배치하는 게 좋습니다.

3. **기술 나열이 이력서와 중복**  
   - 자기소개서 후반부의 기술 스택 나열(React, TS, RN 등)은 포트폴리오/이력서에 이미 충분히 나와 있습니다.  
   - 자기소개서에서는 “기술을 통해 어떤 문제를 해결했는지”에 더 집중하는 것이 좋습니다.

4. **문장 길이가 길고 호흡이 일정치 않음**  
   - 한 문장에 정보가 너무 많이 들어가 있어 읽는 사람이 중간에 놓치기 쉽습니다.  
   - 특히 성과/숫자/도구가 한 문장에 섞인 부분은 2~3문장으로 나누는 편이 좋습니다.

5. **장점과 단점의 밸런스**  
   - 꼼꼼함과 우선순위 조절 이야기는 좋지만, “이걸 어떻게 개선했고 지금은 어느 정도 해결했는지”가 더 구체적이면 좋습니다.  
   - 예: 실제로 우선순위 정리 도구(칸반, Jira, 아이젠하워 매트릭스 등)를 활용했다는 사례를 간단히 넣을 수 있습니다.

### 1-3. 바로 반영할 수 있는 문장/구조 수정 제안

#### (1) 도입부 정리 예시

**현재 느낌**: 성장 스토리는 좋으나 약간 서술형 에세이 느낌.  
**제안**:

```text
어릴 때부터 컴퓨터를 좋아했지만, 단순히 게임을 하는 것보다 “이 화면 뒤에서는 무슨 일이 벌어질까?”를 더 궁금해하는 타입이었습니다. 대학교에서 컴퓨터공학을 전공하면서 웹과 모바일 개발 수업을 들었고, 사용자가 직접 눈으로 보고 손으로 만지는 제품을 만든다는 점에 매력을 느껴 웹/앱 개발자를 진로로 정하게 되었습니다.
```

→ 다음처럼 압축:

```text
저는 “화면 뒤에서 무슨 일이 일어나는지”를 궁금해하며 자란 개발자입니다. 
컴퓨터공학 전공 시절 웹·모바일 개발 수업을 들으며, 사용자가 직접 눈으로 보고 손으로 만지는 제품을 만드는 일에 매력을 느껴 웹/앱 프론트엔드 개발자를 진로로 정했습니다.
```

#### (2) 기술 나열 부분을 “문제 해결 중심”으로 변경

**현재**:

```text
기술적으로는 현재 React, TypeScript, React Query, Next.js 등을 주로 사용하고 있으며, 간단한 서버 작업은 Node.js(Express, NestJS 기초)를 활용할 수 있습니다. 모바일 쪽은 React Native를 기반으로 화면 개발과 API 연동, 기본적인 네이티브 모듈 연동 경험을 가지고 있습니다.
```

**제안**:

```text
최근에는 React, TypeScript, React Query, Next.js를 주로 활용해 성능과 유지보수성을 함께 고려한 웹 서비스를 개발했습니다. 
또한 React Native 기반으로 화면 개발과 API 연동, 푸시 알림 등 네이티브 모듈 연동을 경험하며, 웹에서 검증된 기능을 모바일 환경에 맞게 옮기는 작업을 수행했습니다.
간단한 서버 작업은 Node.js(Express, NestJS 기초)를 활용해 직접 REST API를 설계·구현한 경험이 있습니다.
```

#### (3) 지원 회사 맞춤 문단 추가 예시

지원하려는 회사가 예를 들어 “B2C 구독형 서비스”라고 가정하면:

```text
특히 귀사가 운영하는 [서비스명]처럼, 반복적으로 사용하는 B2C 서비스에서는 작은 성능 차이와 UX 디테일이 곧 이탈률과 전환율로 이어진다고 생각합니다. 
이전 회사에서 랜딩 페이지의 LCP를 3.2초에서 1.8초로 개선하며 이탈률을 약 15% 줄였던 경험을 바탕으로, 귀사의 핵심 지표 개선에도 기여하고 싶습니다.
```

→ 실제 지원 시에는 [서비스명], “구독”, “커머스”, “핀테크” 등 도메인 키워드를 바꿔 넣으면 됩니다.

## 2. 포트폴리오 피드백

### 2-1. 내용/구조/스토리텔링 개선 아이디어

#### (1) 내용 측면

1. **“문제–행동–결과” 구조를 더 명확히**  
   - 이미 일부 프로젝트에 잘 쓰고 있지만, 모든 주요 프로젝트에 통일해서 적용하면 좋습니다.  
   - 예:  
     - 문제: 마케팅 랜딩 페이지 초기 로딩이 느려 이탈률이 높음  
     - 행동: 이미지 최적화, 코드 스플리팅, 번들 다이어트, 불필요 라이브러리 제거  
     - 결과: LCP 3.2s → 1.8s, 이탈률 15% 감소

2. **각 프로젝트에 “내 역할 비중” 명시**  
   - 팀 프로젝트에서 본인이 30%/50%/핵심 기능 담당 등 어느 정도를 맡았는지 짧게라도 써주면 좋습니다.  
   - 예: “프론트엔드 전체 중 약 60% 구현 담당 (로그인/회원가입, 마이페이지, 공통 레이아웃 등)”

3. **기술 선택 이유/트레이드오프 간단히 추가**  
   - React Query, Zustand, Redux Toolkit 등 다양한 상태관리 도구를 써봤다면, 각 프로젝트에서 왜 그 도구를 선택했는지 한 줄씩만 적어도 시니어 면접관에게 좋은 인상을 줍니다.

#### (2) 구조 측면

1. **상단에 “요약 프로필 + 핵심 3포인트” 섹션 추가**  
   - 예:  
     - “웹·모바일 모두 경험한 React 기반 프론트엔드 개발자”  
     - “레거시 → React SPA 마이그레이션, LCP 40% 개선 경험”  
     - “주간 데모/기술 공유로 협업 문화 개선 경험”

2. **“웹 / 모바일 / 백엔드 기초 / 협업”으로 섹션 분리**  
   - 지금도 나뉘어 있지만, 시각적으로 더 명확하게 구분하면 좋습니다.  
   - 예: 포트폴리오 사이트라면 탭 또는 앵커 링크로 “Web Projects / Mobile Apps / Side Projects / Talks & Blog” 등.

3. **사이드 프로젝트를 “실제 서비스처럼” 정리**  
   - “Habitly”나 To-Do 앱도 다음 구조로:  
     - 서비스 한 줄 설명  
     - 타깃 사용자/문제  
     - 주요 기능 (3~5개)  
     - 사용 기술 + 이유  
     - 배포 링크 / GitHub / 회고 링크

#### (3) 스토리텔링 측면

1. **“레거시 → 모던 프론트엔드 → 모바일 → 제품 관점”이라는 성장 축 강조**  
   - 본인의 커리어는 자연스럽게 “기술 스택 확장 + 사용자 접점 확대”라는 흐름이 있습니다.  
   - 포트폴리오 소개 글에서 이 흐름을 3~4문장으로 요약하면 좋습니다.

2. **“사용자 경험 개선” 사례를 하나의 스토리로 묶기**  
   - 예:  
     - 레거시 SPA 마이그레이션에서 페이지 전환 속도 개선  
     - 모바일 앱에서 로그인/회원가입 크래시 감소  
     - 사이드 프로젝트에서 모바일 중심 UI 설계  
   - 이를 “UX 개선 사례 모음” 섹션으로 묶으면, “UX에 진심인 개발자”라는 인상을 줄 수 있습니다.

3. **“협업 문화 기여”를 별도 섹션으로**  
   - 주간 데모, 성능 개선 가이드, 배포 체크리스트, 기술 블로그/스터디 등은 하나의 강력한 테마입니다.  
   - “Team & Culture” 섹션을 만들어,  
     - 어떤 문제를 보고  
     - 어떤 제안을 했고  
     - 팀이 어떻게 변했는지  
     를 2~3개 사례로 정리하면 좋습니다.

### 2-2. 부족한 증빙 자료나 추가 산출물 제안

1. **GitHub 리포지토리 및 README 강화**  
   - Habitly, To-Do 앱, 프리랜서 랜딩 페이지 등 각 프로젝트의 README에  
     - 프로젝트 소개  
     - 기술 스택  
     - 주요 기능 GIF/스크린샷  
     - 아키텍처/디렉토리 구조  
     - 배운 점/회고 링크  
     를 추가하면, 포트폴리오 신뢰도가 크게 올라갑니다.  
   - 검색 키워드 예: `GitHub README portfolio examples`, `awesome readme`

2. **성능 개선 전/후 지표 캡처**  
   - Lighthouse 리포트 스크린샷, LCP/CLS/TTI 전후 비교 그래프 등을 이미지로 첨부하면 “숫자”가 시각적으로 전달됩니다.  
   - 검색 키워드 예: `Lighthouse report before after performance case study`

3. **스토리북 / 디자인 시스템 데모 링크**  
   - 네오픽셀랩에서 만든 공통 컴포넌트/디자인 시스템이 있다면, 가능 범위 내에서 비식별화한 샘플 스토리북이나 코드 스니펫을 정리해두면 좋습니다.  
   - 검색 키워드 예: `Storybook design system case study`, `React component library portfolio`

4. **기술 블로그 글 링크 모음**  
   - 포트폴리오에 “Top 3 Posts” 정도로 링크를 노출하면, 깊이 있는 사고와 글쓰기 역량을 보여줄 수 있습니다.  
   - 예:  
     - “React SPA 마이그레이션 회고”  
     - “React Native 로그인 플로우 안정화 과정”  
     - “프론트엔드 성능 개선 체크리스트”

5. **데모 영상(짧은 Loom/YouTube)**  
   - 웹/앱 사용 흐름을 1~3분으로 녹화해 링크를 걸어두면, 면접관이 빠르게 이해할 수 있습니다.  
   - 검색 키워드 예: `Loom product demo example`, `portfolio project walkthrough video`

## 4. 답변 기반 보완 활동 추천

### 4-1. 면접 답변에서 드러난 부족한 경험/역량

1. **답변 구조화·논리화 부족**  
   - 1번, 2번 답변은 핵심 메시지는 있으나 문장이 흐릿하고, “문제–행동–결과”가 명확히 분리되어 있지 않습니다.  
   - “에스피에이”, “리트기반해” 등 오타/발음 기반 표기가 많아 실제 면접에서도 말이 꼬이거나 긴장도가 높을 가능성이 보입니다.

2. **전문적인 상황에서도 감정/언어 통제 미흡 (3번 답변)**  
   - 3번 답변은 욕설과 혼잣말이 섞여 있어, 실제 면접 상황에서는 치명적입니다.  
   - 스트레스 상황에서의 감정 조절, 말하기 연습이 필요해 보입니다.

3. **핵심 메시지 요약 능력 부족**  
   - “이 경험에서 무엇을 배웠는지”를 한 문장으로 정리하는 부분이 약해 보입니다.  
   - 면접관이 듣고 싶은 “결론”이 뒤늦게 나오거나, 아예 나오지 않을 수 있습니다.

4. **도메인/비즈니스 관점 언급 부족**  
   - 기술/구현 관점은 강하지만, “왜 이 기능이 비즈니스에 중요했는지”에 대한 언급이 상대적으로 적습니다.

### 4-2. 이를 채울 수 있는 구체적 활동 아이디어

#### 1) 모의 면접 스터디 참여 (온라인/오프라인)

- **내용**: 프론트엔드/웹 개발자 대상 모의 면접 스터디에 참여해, 실제 면접 질문(프로젝트 설명, 성능 개선, 협업 갈등 등)에 대해 2~3분 내로 말하는 연습을 반복합니다.  
- **예상 검색 키워드**: `프론트엔드 모의면접 스터디`, `개발자 면접 스터디 서울`, `React 개발자 면접 질문`  
- **추천 이유**: 실제 사람 앞에서 말해보며 긴장감·시간 관리·언어 선택을 동시에 훈련할 수 있고, 피드백을 통해 답변 구조를 개선할 수 있습니다.

#### 2) “프로젝트 회고 발표”를 정기적으로 진행

- **내용**: 현재/과거 프로젝트(레거시 마이그레이션, RN 앱, Habitly 등)를 주제로, 10분 발표 자료를 만들어 스터디/커뮤니티에서 발표합니다.  
- **예상 검색 키워드**: `프론트엔드 스터디 발표`, `React 스터디 모임 발표`, `기술 발표 회고`  
- **추천 이유**: 슬라이드를 만들면서 자연스럽게 “문제–해결–결과–배운 점” 구조를 정리하게 되고, 발표를 통해 말하기/논리 전개 능력이 향상됩니다.

#### 3) 비개발자에게 설명하는 연습 (블로그/영상)

- **내용**: 본인의 성능 개선, SPA 전환, RN 로그인 플로우 개선 경험을 “개발을 모르는 사람도 이해할 수 있게” 블로그 글이나 짧은 영상으로 설명해봅니다.  
- **예상 검색 키워드**: `기술 블로그 글쓰기 팁`, `개발 비전공자에게 설명하기`, `Loom screen recording tutorial`  
- **추천 이유**: 전문 용어를 줄이고 핵심만 추리는 연습이 되어, 면접에서 “짧고 명확한 답변”을 하는 데 큰 도움이 됩니다.

#### 4) 감정/언어 컨트롤을 위한 “녹음·피드백 루프”

- **내용**: 예상 면접 질문(예: “가장 어려웠던 프로젝트는?”, “협업 갈등 경험은?”) 10개를 뽑아, 답변을 음성으로 녹음한 뒤 스스로 혹은 지인과 함께 들어보며 욕설, 불필요한 추임새, 반복 표현을 체크합니다.  
- **예상 검색 키워드**: `개발자 면접 예상 질문`, `자기소개 녹음 피드백`, `스피치 트레이닝 앱`  
- **추천 이유**: 3번 답변처럼 감정이 섞일 때 나오는 언어 습관을 사전에 인지하고 교정할 수 있습니다.

#### 5) 비즈니스/제품 관점 강화 프로젝트

- **내용**: 사이드 프로젝트나 프리랜서 작업에서, 단순 기능 구현이 아닌 “지표 개선 목표”를 먼저 설정하고 진행합니다.  
  - 예: Habitly에 “7일 연속 사용률 20% 향상”을 목표로 리마인드 알림, 온보딩 개선, 간단한 A/B 테스트를 설계.  
- **예상 검색 키워드**: `제품 지표 설정 방법`, `UX 개선 A/B 테스트`, `Firebase Analytics React Native`  
- **추천 이유**: 면접에서 “이 기능이 왜 중요한지, 어떤 지표를 개선했는지”를 자연스럽게 말할 수 있게 됩니다.

#### 6) 오픈소스/커뮤니티 코드 리뷰 참여

- **내용**: React/React Native 관련 오픈소스 이슈나 국내 FE 커뮤니티에서 코드 리뷰를 주고받는 활동을 합니다.  
- **예상 검색 키워드**: `React open source good first issue`, `프론트엔드 코드리뷰 커뮤니티`, `FEConf Slack`  
- **추천 이유**: 다른 사람의 코드를 설명하고 피드백하는 과정에서 논리적 표현과 기술적 커뮤니케이션 능력이 향상됩니다. 면접에서도 “코드 리뷰 경험”을 구체적으로 말할 수 있습니다.

#### 7) 실전 면접 특화 강의/코칭 활용

- **내용**: 개발자 면접 특화 온라인 강의(예: 인프런 “개발자 면접 완전 정복”, 패스트캠퍼스 커리어 강의 등)를 수강하고, 제공되는 템플릿에 맞춰 답변 스크립트를 작성해봅니다.  
- **예상 검색 키워드**: `인프런 개발자 면접`, `프론트엔드 면접 대비 강의`, `개발자 커리어 코칭`  
- **추천 이유**: 예상 질문 리스트와 모범 답안을 참고해 자신의 경험을 구조화할 수 있고, 1·2번 답변의 구조를 더 탄탄하게 만들 수 있습니다.

### 마무리 제안

1. **우선순위**  
   - 1단계: 자기소개서/포트폴리오에서 이미 잘하고 있는 “구체적 성과”를 유지하되, 문장 구조와 지원 회사 맞춤형 문장을 보완.  
   - 2단계: 모의 면접 스터디 + 녹음·피드백을 통해 말하기 구조와 언어 습관 교정.  
   - 3단계: 제품/지표 관점이 드러나는 사이드 프로젝트 개선 및 블로그/발표로 스토리텔링 강화.

2. **실행 플랜 예시 (4주)**  
   - 1주차: 자기소개서/포트폴리오 리팩터링 + GitHub README/스크린샷 정리  
   - 2주차: 모의 면접 스터디 1회 참여 + 예상 질문 10개 녹음·피드백  
   - 3주차: Habitly 또는 RN 앱에 “지표 개선 목표”를 하나 설정하고 기능 개선  
   - 4주차: 개선 내용 회고 블로그 작성 + 스터디/커뮤니티에서 10분 발표

이 정도만 실행해도, 현재 이미 탄탄한 실무 경험 위에 “말하기·스토리텔링·제품 관점”이 더해져 면접 경쟁력이 크게 올라갈 것입니다.
