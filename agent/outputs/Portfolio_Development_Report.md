# 문서 피드백 및 성장 제안

> 참고: 실제 웹 검색은 실패했지만, 2024년까지의 업계 동향과 일반적으로 알려진 교육/프로그램을 기반으로, “있을 법한” 출처와 검색 키워드를 함께 제안합니다.

---

## 1. 자기소개서 피드백

### 1-1. 강점

1. **커리어 스토리가 자연스럽다**  
   - “어릴 때 컴퓨터 → 전공 → 웹/모바일 수업 → 스타트업 → 모바일 회사 → 프리랜서/사이드 프로젝트”로 이어지는 흐름이 좋아, 읽는 사람이 성장 경로를 쉽게 이해할 수 있습니다.

2. **웹 + 모바일 경험이 균형 있게 드러난다**  
   - React / React Native, 웹 SPA 마이그레이션, 모바일 앱 신규 개발 등 “풀프론트엔드” 포지션에 적합한 경험이 잘 정리돼 있습니다.

3. **협업/커뮤니케이션에 대한 태도가 구체적이다**  
   - “주간 데모 세션을 제안해 운영했다”, “기획 의도를 더 명확히 이해했다” 등 행동과 결과가 있어 신뢰도가 높습니다.

4. **성능·UX·유지보수성을 같이 보는 시각**  
   - LCP 개선, 코드 구조 고민, 디자인 시스템 등 단순 기능 구현을 넘어선 고민이 드러나 강점입니다.

5. **자기 성찰과 개선 의지가 보인다**  
   - 꼼꼼함이 장점이면서도, 우선순위 조절을 의식적으로 한다는 부분은 “자기 인식이 있는 사람”이라는 인상을 줍니다.

### 1-2. 개선 포인트

1. **문단이 길고, 핵심 메시지가 분산되어 있다**  
   - 채용 담당자가 빠르게 스캔하기 어렵습니다. “핵심 성과 한 줄 → 세부 설명” 구조로 재구성하면 좋습니다.

2. **정량적 성과를 더 앞쪽에 배치하면 좋다**  
   - LCP 개선, 이탈률 감소, 다운로드 수 등 좋은 숫자가 뒤에 있어 눈에 잘 안 들어옵니다.

3. **지원 회사/직무에 대한 ‘맞춤형’ 언급이 부족하다**  
   - “귀사”라고만 하고, 해당 회사의 서비스 특성이나 기술 스택에 맞춘 구체적인 연결이 없습니다.

4. **반복되는 표현이 많다**  
   - “웹/앱 개발자”, “사용자 경험”, “협업을 중요하게 생각” 등이 반복되어 문장이 길어집니다. 중복을 줄이고 더 강한 키워드를 남기면 좋습니다.

5. **마지막 문단이 다소 추상적이다**  
   - “성장을 돕겠다, 기여하겠다”는 문장보다, “입사 후 6개월 내에 ~을 하겠다”처럼 구체적인 그림이 있으면 더 설득력 있습니다.

### 1-3. 바로 반영할 수 있는 문장/구조 수정 제안

#### (1) 도입부 정리

**현재**  
> 어릴 때부터 컴퓨터를 좋아했지만, 단순히 게임을 하는 것보다 “이 화면 뒤에서는 무슨 일이 벌어질까?”를 더 궁금해하는 타입이었습니다. …

**제안**  
```text
저는 웹과 모바일 환경 모두에서 사용자 경험을 개선해 온 프론트엔드 개발자입니다.
어릴 때부터 “이 화면 뒤에서는 무슨 일이 벌어질까?”를 궁금해했고, 컴퓨터공학 전공과
이후 웹/모바일 실무 경험을 통해 그 궁금증을 실제 서비스로 풀어 왔습니다.
```

#### (2) 학부·초기 경력 부분 압축

**현재**: HTML/CSS/JS, React, Node.js 학습 이야기가 길게 서술형으로 나열  

**제안**  
```text
학부 시절에는 HTML/CSS, JavaScript를 기반으로 간단한 웹사이트를 만들며
프론트엔드 기초를 다졌고, 이후 React와 TypeScript로 팀 프로젝트를 진행하며
컴포넌트 설계와 상태 관리에 대한 이해를 쌓았습니다. 동시에 Node.js(Express)와
PostgreSQL/MongoDB로 간단한 REST API를 구현하며, 프론트엔드와 서버 간
데이터 흐름을 함께 고려하는 습관을 갖게 되었습니다.
```

#### (3) 성과를 앞으로 끌어오기

```text
첫 직장인 스타트업에서는 jQuery 기반 레거시 웹을 React SPA로 마이그레이션하며
코드 중복을 줄이고 신규 기능 개발 속도를 약 20% 단축했습니다.
또한 마케팅 페이지의 LCP를 3.2초에서 1.8초로 개선해 이탈률을 약 15% 낮추었고,
이 경험을 바탕으로 팀 내 '프론트엔드 성능 개선 가이드'를 작성해 공유했습니다.
```

#### (4) 모바일 경험 요약 개선

```text
이후 React Native 기반 모바일 앱 개발사로 이직해, 기존 웹 서비스를 모바일 앱으로
확장하는 프로젝트에 참여했습니다. 로그인/회원가입, 푸시 알림, 채팅 화면 등
핵심 사용자 플로우를 구현했고, iOS/Android 스토어 배포와 크래시 리포트 분석까지
서비스 운영 전 과정을 경험했습니다. 이 과정에서 모바일 환경에서는 같은 기능도
더 단순하고 명확한 UX가 필요하다는 점을 체감했습니다.
```

#### (5) 마무리 문단 구체화

```text
입사 후 6개월 안에는 귀사의 주요 웹/앱 화면을 분석해 성능과 UX 개선 우선순위를
정리하고, 실제로 LCP·전환율 등 핵심 지표를 개선하는 데 기여하고 싶습니다.
또한 제가 경험한 React/React Native 기반 개발 노하우를 팀 위키와 기술 공유
세션으로 정리해, 동료들과 함께 성장하는 문화를 만드는 데 힘을 보태겠습니다.
```

## 2. 포트폴리오 피드백

### 2-1. 내용/구조/스토리텔링 개선 아이디어

#### (1) 내용 측면

1. **프로젝트별 “한 줄 임팩트 문장” 추가**  
   - 각 프로젝트 시작 부분에 “이 프로젝트에서 내가 한 가장 중요한 일 1가지”를 한 줄로 요약하면 좋습니다.  
   - 예: `레거시 웹을 React SPA로 전환해, 신규 기능 개발 속도를 20% 단축했습니다.`

2. **기술 스택을 “깊이” 기준으로 재정리**  
   - 지금은 나열형이라, 무엇을 “실무 주력”으로 쓰는지 애매합니다.  
   - 예: `주력(실무 1년 이상): React, TypeScript, React Query, Next.js, React Native` / `사용 경험: NestJS, PostgreSQL, MongoDB`처럼 레벨을 명시.

3. **프리랜서/사이드 프로젝트의 비즈니스 맥락 보강**  
   - “SaaS 랜딩 페이지”가 어떤 타깃, 어떤 목표(리드 수집, 베타 신청 등)를 위해 만들어졌는지 한 줄씩 추가하면 좋습니다.

#### (2) 구조 측면

1. **상단에 “요약 섹션” 추가**  
   - 이력서처럼, 포트폴리오 최상단에 `프로필 · 핵심 스킬 · 대표 프로젝트 3개`만 요약한 섹션을 두면, 바쁜 리크루터가 핵심을 빠르게 파악할 수 있습니다.

2. **웹 / 모바일 / 공통 프로젝트로 카테고리화**  
   - 지금은 시간 순이라 강점이 섞여 있습니다.  
   - 예:  
     - Web Projects (React/Next.js)  
     - Mobile Projects (React Native)  
     - Common: 디자인 시스템, 성능 개선, 협업/문서화

3. **“성과/지표”를 한 눈에 보이게 정리**  
   - 별도 섹션으로 `Impact & Metrics`를 두고,  
     - LCP 3.2s → 1.8s  
     - 이탈률 15% 감소  
     - 앱 다운로드 5만+  
     - 크래시율 40% 감소  
     등을 bullet로 정리하면 좋습니다.

#### (3) 스토리텔링 측면

1. **“문제 → 행동 → 결과” 구조를 일관되게**  
   - 일부 프로젝트는 잘 되어 있지만, 일부는 “내용 나열”에 그칩니다.  
   - 예:  
     - 문제: 마케팅 페이지 초기 로딩이 느려 이탈률이 높음  
     - 행동: 이미지 최적화, 코드 스플리팅, 라이브러리 정리  
     - 결과: LCP 3.2s → 1.8s, 이탈률 15% 감소

2. **“내가 없었으면 안 됐을 부분”을 강조**  
   - 팀 프로젝트에서 본인의 역할이 겹쳐 보일 수 있으므로,  
     - 내가 주도한 의사결정  
     - 내가 처음 도입한 도구/프로세스  
     등을 더 분명히 쓰면 좋습니다.  
   - 예: `Storybook 도입을 제안하고 구축까지 담당` 등.

3. **커리어 방향성과 프로젝트를 연결**  
   - “사용자의 문제를 이해하고 해결하는 개발자”가 목표라면, 각 프로젝트에서  
     - 어떤 사용자 문제를 정의했고  
     - 어떻게 해결했는지  
     를 1~2줄씩 붙이면, 스토리가 더 강해집니다.

### 2-2. 부족한 증빙 자료 / 추가 산출물 제안

1. **실제 화면/데모 링크**  
   - 랜딩 페이지, Habitly, To-Do 앱의  
     - 배포 URL (웹)  
     - 앱 스토어/테스트플라이트 링크 (가능하다면)  
     - 또는 짧은 GIF/스크린샷  
   - → “코드만 있는 프로젝트”보다 설득력이 훨씬 커집니다.

2. **기술 블로그 글 링크**  
   - 이미 블로그를 운영하신다고 했으니,  
     - 레거시 → React 마이그레이션 회고  
     - 성능 개선(Lighthouse/LCP) 실험기  
     - React Native 크래시 줄이기 사례  
   - 이런 글을 각 프로젝트 하단에 “관련 포스팅”으로 연결하면 좋습니다.

3. **디자인 시스템/컴포넌트 라이브러리 문서**  
   - Storybook 링크나, 디자인 시스템 가이드 문서(캡처라도)를 첨부하면,  
     - “디자인 시스템 구축 경험 있음”이 단순 문장이 아니라 실제 산출물로 증명됩니다.

4. **테스트/품질 관련 산출물**  
   - Jest/React Testing Library 테스트 코드 일부,  
   - 배포 체크리스트 문서,  
   - Crashlytics 이슈 정리 문서(민감 정보 제거 후)  
   - → “꼼꼼함 + 운영 경험”을 뒷받침합니다.

5. **협업 프로세스 예시**  
   - Figma → 구현까지의 핸드오프 방식,  
   - Jira 이슈 템플릿,  
   - 주간 데모 세션 아젠다 캡처  
   - → “협업을 중요하게 생각한다”는 말을 구체화해 줍니다.

## 4. 답변 기반 보완 활동 추천

### 4-1. 면접 답변에서 부족해 보이는 경험/역량

1. **1번 답변 – 기술/성과는 있으나 구조와 표현이 흐릿함**  
   - “리트, 에스피에이, 컨포란트, 코트” 등 오타가 많고,  
   - “어떻게, 얼마나 개선했는지”가 자기소개서/포트폴리오만큼 명확하게 드러나지 않습니다.  
   - → 면접에서 말로 설명할 때 **구조화된 스토리텔링**과 **정확한 용어 사용**이 약해 보입니다.

2. **2번 답변 – 개념은 맞지만 예시가 더 구체적일 수 있음**  
   - 모바일/웹 UX 차이에 대한 이해는 있으나,  
   - “내가 실제 프로젝트에서 어떻게 적용했는지” 사례가 부족합니다.

3. **3번 답변 – 부적절한 언어 사용 및 감정 통제 실패**  
   - 욕설과 비논리적인 문장이 섞여 있어,  
   - 실제라면 “스트레스 상황에서의 태도”, “프로페셔널리즘”에 큰 마이너스입니다.  
   - → 감정 관리, 말하기 훈련, 면접 상황 시뮬레이션이 필요해 보입니다.

### 4-2. 이를 채울 수 있는 구체적 활동 아이디어 (5개 이상)

#### 1) “프로젝트 설명 스크립트” 작성 & 모의 면접

- **내용**:  
  - 각 대표 프로젝트(레거시 마이그레이션, 성능 개선, RN 앱 개발)에 대해  
    - 문제 → 내 역할 → 기술 선택 이유 → 구현 방식 → 성과  
    를 A4 1장 이내로 정리하고, 이를 바탕으로 친구/스터디원과 모의 면접 진행.
- **예상 검색 키워드**: `프론트엔드 면접 질문 리스트`, `React 개발자 프로젝트 설명 예시`, `개발자 모의 면접 스터디`  
- **추천 이유**:  
  - 1번 답변처럼 흐릿하게 말하는 습관을 줄이고, **구조화된 말하기**를 몸에 익힐 수 있습니다.

#### 2) “모바일 vs 웹 UX 차이”를 주제로 블로그 글/발표 준비

- **내용**:  
  - 실제 본인이 만든 웹 화면 1개, 모바일 화면 1개를 골라  
    - 정보 구조, 내비게이션, 입력 방식, 네트워크 이슈 대응  
    관점에서 비교 분석 글 작성 후, 스터디/슬랙 등에서 발표.
- **예상 검색 키워드**: `모바일 웹 UX 차이 사례`, `React Native 모바일 UX 베스트 프랙티스`  
- **추천 이유**:  
  - 2번 답변을 더 풍부한 사례와 용어로 설명할 수 있게 되고,  
  - “UX를 고민하는 개발자” 이미지를 강화할 수 있습니다.

#### 3) 감정/스트레스 관리 + 말하기 훈련 (코칭/강의/동아리 활용)

- **내용**:  
  - 스피치/프레젠테이션 강의 또는 Toastmasters 같은 스피치 클럽,  
  - 혹은 커리어 코칭/상담을 통해,  
    - 욕설/부정적 표현을 통제하고  
    - 긴장 상황에서도 논리적으로 말하는 연습.
- **예상 검색 키워드**: `개발자 발표 스피치 강의`, `Toastmasters 서울`, `직장인 커리어 코칭 상담`  
- **추천 이유**:  
  - 3번 답변 수준의 발화는 실제 면접에서 치명적이므로,  
  - **프로페셔널한 언어 습관**을 만드는 것이 매우 중요합니다.

#### 4) “성능 개선 미니 프로젝트” 진행 및 공개

- **내용**:  
  - 간단한 React/Next.js 앱을 만들어, Lighthouse로 성능 측정 후  
    - 이미지 최적화, 코드 스플리팅, React.lazy, prefetch 등 적용 전/후 지표 비교.  
  - 결과를 GitHub + 블로그에 정리.
- **예상 검색 키워드**: `Next.js 성능 최적화 LCP 개선 예제`, `React 코드 스플리팅 실무`, `web.dev LCP optimization`  
- **추천 이유**:  
  - 이미 성능 개선 경험이 있지만,  
  - **최신 Next.js/React Query/SSR 환경에서의 성능 튜닝** 경험을 추가하면 2025년 기준 시장성과 면접 답변의 깊이가 올라갑니다.

#### 5) React Native 실전 앱 1개 추가 (스토어 배포까지)

- **내용**:  
  - 실제 사용자를 상정한 작은 앱(예: 일정 공유, 간단 커뮤니티, 로컬 알림 기반 습관 앱)을 기획 → RN(Expo + TS)로 개발 → Play Store/앱스토어 또는 최소한 TestFlight/내부 배포까지 진행.  
  - Crashlytics/Sentry 연동, OTA 업데이트(Expo EAS)까지 시도.
- **예상 검색 키워드**: `React Native Expo EAS 배포 튜토리얼`, `React Native Firebase Crashlytics 연동`, `React Native 사이드 프로젝트 아이디어`  
- **추천 이유**:  
  - 모바일 운영 경험이 이미 있지만,  
  - **본인이 처음부터 끝까지 주도한 RN 앱**이 하나 더 있으면, 모바일 포지션/하이브리드 포지션 모두에 강점을 가질 수 있습니다.

#### 6) 오픈소스/커뮤니티 기여 (작은 이슈부터)

- **내용**:  
  - React, React Query, React Native 관련 오픈소스의 문서 수정, 예제 코드 개선, 이슈 재현/정리 등 작은 기여부터 시작.  
  - 또는 국내 프론트엔드 스터디/슬랙 커뮤니티에서 Q&A/발표 참여.
- **예상 검색 키워드**: `React Query GitHub issues good first issue`, `한국 프론트엔드 개발자 슬랙 커뮤니티`, `오픈소스 기여 시작하기`  
- **추천 이유**:  
  - 협업/커뮤니케이션 역량을 실제 코드 리뷰/논의로 보여줄 수 있고,  
  - “단순 회사 경험”을 넘어 **커뮤니티 활동이 있는 개발자**로 포지셔닝할 수 있습니다.

#### 7) “시스템 설계 관점”의 프론트엔드 학습

- **내용**:  
  - 로그인/회원가입, 채팅, 알림 등 이미 구현한 기능을  
    - API 설계, 캐싱 전략(React Query), 에러 처리, 확장성  
    관점에서 다시 설계해 보는 미니 설계 문서 작성.  
  - 이를 블로그나 포트폴리오에 “미니 시스템 설계”로 추가.
- **예상 검색 키워드**: `프론트엔드 시스템 설계 면접`, `React Query 캐싱 전략`, `프론트엔드 아키텍처 설계 예시`  
- **추천 이유**:  
  - 중급 이상 프론트엔드 면접에서 자주 나오는 “아키텍처/설계” 질문에 대비할 수 있고,  
  - 단순 구현자가 아니라 “구조를 설계하는 개발자” 이미지를 줄 수 있습니다.

이후 단계로는,

1. 위 자기소개서/포트폴리오 수정안을 토대로 **문서 버전업 1차**를 하고,  
2. 대표 프로젝트 2~3개에 대해 **말하기 스크립트**를 만든 뒤,  
3. 모의 면접(지인/스터디/온라인 멘토)을 2~3회 진행해 보시길 권장합니다.

원하시면, 다음 단계로는 “예상 면접 질문 리스트”를 뽑고, 각 질문에 대한 모범 답변 뼈대를 함께 정리해 드리겠습니다.
